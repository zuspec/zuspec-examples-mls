Key observations:
- The constraint version uses 47% of the manual gate count (53% fewer cells).
- This improvement comes from the Cube GROW minimizer exploiting don't-cares:
    38 product terms (GROW) vs 78 (QM) — 51% fewer AND/OR groups.
  The undefined encodings in RV32I (94% of 2048 input minterms) are genuine
  don't-cares; GROW expands each instruction's cube into them without ever
  enumerating them.
- The constraint SOP uses AND/OR/NOT primitives (no MUX cells).
  Each product term is a conjunction that may cover multiple instruction groups.
- The manual case statement maps to MUX chains (37 MUX cells).
  Yosys factors the opcode decode hierarchically, but cannot exploit the
  don't-care space as effectively because case(opcode) locks in the off-set.

Note on the synthesis flow:
The manual always_comb case statement can trigger ROM inference in
yosys (without memory_map, it shows $meminit+$memrd_v2 cells
instead of MUX cells).  memory_map forces decomposition to gates
before techmap for a fair apples-to-apples cell count comparison.

Advantages of the constraint approach:
1. Area: 53% fewer gates due to don't-care exploitation via Cube GROW
2. Extensibility: adding an instruction requires one new block only
3. Correctness: each block is independently verifiable
4. No monolithic decision tree to maintain
5. The GROW minimizer finds prime implicants that span multiple
   instruction groups automatically
