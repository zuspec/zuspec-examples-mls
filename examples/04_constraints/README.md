# Example 04 — Constraints as Design Constructs

## Motivation

Hardware engineers traditionally write instruction decoders as nested `if/else`
trees or `case` statements.  This works, but it has three well-known problems:

1. **Monolithic code.** Adding one instruction touches a central decision tree.
2. **Hard to optimise.** A hand-written case statement usually leaves logic on
   the table; the compiler rarely gets the full picture needed to minimise across
   instruction groups.
3. **Mixed concerns.** Encoding knowledge (what bits select this instruction) and
   microarchitectural semantics (what signals to assert) are tangled together.

Zuspec **constraints** solve all three problems.

---

## The Key Idea

Each RV32I instruction is described by exactly one `@constraint` block:

```python
@constraint
def c_add(self):
    if self.opcode == 0x33 and self.funct3 == 0 and self.funct7b5 == 0:
        assert self.alu_op    == ALU_ADD
        assert self.imm_sel   == IMM_NONE
        assert self.use_rs1   == 1
        assert self.use_rs2   == 1
        assert self.use_rd    == 1
        assert self.is_load   == 0
        assert self.is_store  == 0
        assert self.is_branch == 0
        assert self.is_jal    == 0
        assert self.is_jalr   == 0
```

The `if` guard is a **constraint implication**: when the bits match, the
outputs are forced.  No other block needs to be modified.  Adding `c_sub`
is just adding another block.

The guard uses **named derived fields** (`self.opcode`, `self.funct3`,
`self.funct7b5`) rather than raw bit slices.  The synthesiser's pre-pass
maps each named field back to the corresponding bit range of `instr`, so the
generated RTL is still operating directly on bits.

## Running the Example

### Prerequisites

```bash
pip install yowasp-yosys   # only needed for synthesis comparison
```

The zuspec packages (zuspec-dataclasses, zuspec-synth) are installed via the
workspace's `ivpm.yaml`.

### Using the zuspec CLI

With `zuspec-cli`, `zuspec-fe-sv`, and `zuspec-be-sv` installed:

```bash
cd examples/04_constraints

# Synthesize from SystemVerilog source
zuspec synth --fe sv rv32_pkg.sv --top rv32i_decode_c -o out/rv32i_decode.sv

# Or using the YAML pipeline (all options in one file)
zuspec pipeline pipeline.yaml rv32_pkg.sv
```

The `pipeline.yaml` file in this directory pre-configures the frontend, all
optimization transforms, and the RTL backend.

### Synthesis comparison

```bash
cd examples/04_constraints
python synth_compare.py         # synthesize both versions, print table
python synth_compare.py --write-sv  # also save expected/rv32i_decode.sv
```

### Simulation tests

```bash
cd examples/04_constraints
python -m pytest test_rv32i_decode.py -v
```

## SystemVerilog Version

The same decode logic is available as SystemVerilog action classes.  The SV
sources are annotated with `(* zsp_output *)` attributes and `zsp_mark#(T)::valid()`
calls — the `zuspec-fe-sv` front-end reads these to reconstruct the same IR
that the Python decorator path produces.

### SV synthesis pipeline

```bash
cd examples/04_constraints
python synth_compare_sv.py          # parse SV, synthesize, print emitted RTL
python synth_compare_sv.py --no-synth  # skip cube minimization, show support bits
```

The SV and Python versions produce equivalent constraint-block tables and
identical minimized RTL output.

### SV simulation + Verilator lint

```bash
cd examples/04_constraints
python -m pytest test_rv32i_sv_decode.py -v
```

The test file:
1. Parses `rv32_pkg.sv` through `zuspec-fe-sv` (pyslang-based)
2. Runs the SV → IR → synthesis pipeline
3. Evaluates all RV32I instruction encodings against the constraint-block IR
4. Wraps the emitted SV wire assignments in a module and runs `verilator --lint-only`

All 37 tests pass (36 decode correctness + 1 Verilator lint).

## Results

Synthesis was run with `yowasp-yosys` using
`proc; opt; memory_map; opt; techmap; opt` (generic cell library, no ABC
technology mapping).  `memory_map` is required to decompose the ROM that
yosys infers for `always_comb case` constant assignments, giving a
fair apples-to-apples gate comparison.

```
==============================================================
  RV32I Decode Synthesis Comparison
==============================================================
  Metric                    Constraint          Manual
--------------------------------------------------------------
  Total cells                       80             168  ( 47%)
  Wires                             96             104  ( 92%)
  Wire bits                        155             316  ( 49%)
==============================================================

  Cell-type breakdown:
  Type                      Constraint          Manual
--------------------------------------------------------------
  $_AND_                            52              16
  $_MUX_                             0              37
  $_NOT_                             8              27
  $_OR_                             20              88
```

**The constraint version uses 47% of the manual gate count — 53% fewer cells.**
This result comes from the **Cube GROW minimizer** built into `zuspec-synth`.
RV32I has 2048 possible input encodings but only 121 are defined by instructions
— 94% genuine don't-cares.  GROW expands each instruction's cube into that
don't-care space, finding prime implicants that span multiple instructions and
don't-care regions simultaneously.  This is impossible with a hand-coded
`case` statement because the switch arms lock the off-set.

The structural difference:
- **Constraint SOP**: 52 AND + 8 NOT + 20 OR (flat sum-of-products, no MUX).
  38 prime implicants (vs 78 from QM alone — 51% fewer), many shared across
  outputs via CSE.  Each PI may cover multiple instruction groups.
- **Manual case statement**: 16 AND + 37 MUX + 27 NOT + 88 OR (hierarchical).
  Yosys factors the opcode decode once but cannot exploit the don't-care space,
  resulting in 2× more cells total.

**Advantages of the constraint approach:**
1. **Area: 53% fewer gates** — don't-care exploitation via Cube GROW.
2. Extensibility — adding an instruction is one new block; existing code is
   never touched.
3. Correctness — each block is independently readable and verifiable.
4. No monolithic decision tree — the synthesiser assembles the logic.
5. GROW minimisation is automatic — the engineer writes intent, not gate counts.

## Zero-touch Extensibility

`rv32m_decode.py` shows how to add the RV32M multiply/divide extension:

```python
@zdc.dataclass
class RV32MDecode(RV32IDecode):
    """RV32I + M-extension (multiply/divide)."""

    funct7b0: zdc.u1 = zdc.rand()   # instr[25]

    @constraint
    def c_funct7b0_extraction(self):
        assert self.funct7b0 == ((self.instr >> 25) & 0x1)

    @constraint
    def c_mul(self):
        if self.opcode == 0x33 and self.funct3 == 0 and self.funct7b0 == 1:
            assert self.alu_op  == ALU_MUL
            ...
```

**No line of `rv32i_decode.py` was touched.**  The eight M-extension
constraints simply inherit all 40 base constraints and add their own.

## Teaching Points

1. **One block per instruction.**  Each instruction is self-contained.  The
   decoder is the union of 40 independent blocks, not one monolithic tree.

2. **Synthesiser does the work — better than you could do by hand.**  The Zuspec
   synthesiser runs the Cube GROW minimiser across all blocks simultaneously.
   GROW exploits the 94% don't-care space in RV32I encodings to find prime
   implicants that span multiple instruction groups.  The resulting RTL uses
   **53% fewer cells** than a well-written hand-coded case statement — an
   improvement no human could easily achieve manually.

3. **No external tools required.**  The GROW minimiser is built into
   `zuspec-synth`.  `yowasp-yosys` is only used here to independently
   validate and measure the output.

4. **Constraints are implications.**  `if guard: assert output` is
   *constraint implies*, identical to `guard → output` in formal logic.
   The synthesiser reads every block's guard as an input minterm and every
   asserted field as an output function.

5. **Named derived fields.**  `self.opcode`, `self.funct3`, `self.funct7b5`
   are declared with `zdc.rand()` and bound to bit slices of `instr` by a
   single `@constraint` extraction block.  The synthesiser's pre-pass maps
   every named-field guard back to the original bit range, so the generated
   SV has no redundant muxing.
